# ADR-011: JWT Key Rotation SLA

> **日期**: 2026-05-08
> **决策人**: 架构评审委员会
> **状态**: 已批准

---

## 背景

SchemaPlexAI 使用 RS256 签名的 JWT 进行服务间及用户身份认证。当前 JWT 密钥为静态配置，无定期轮换机制，存在以下风险：

- **合规风险**：SOC2 Type II 审计要求密钥必须在固定周期内轮换
- **泄露窗口**：密钥一旦泄露，在手动干预前长期有效
- **撤销困难**：JWT 无状态特性导致无法单独撤销某个密钥签发的令牌

同时，系统计划集成 HashiCorp Vault 作为统一密钥管理基础设施，需要明确轮换 SLA 以指导 Vault 策略配置。

## 决策

1. **JWT 签名密钥执行 90 天强制轮换 SLA**
   - 自动轮换周期：90 天
   - 宽限期：轮换后旧密钥保留 7 天（兼容已签发但未过期的令牌）
   - 紧急轮换：密钥泄露或安全事件时支持手动即时轮换

2. **JWKS 端点设计**
   - Gateway 暴露 `/.well-known/jwks.json` 端点
   - 返回当前有效密钥集合（含新密钥 + 宽限期内的旧密钥）
   - 所有下游服务通过 JWKS 动态获取公钥，禁止本地硬编码公钥

3. **算法迁移**
   - 当前：RS256（2048-bit RSA）
   - 目标：保持 RS256，密钥长度提升至 3072-bit（2030 年前瞻合规）
   - 轮换时同步完成密钥长度升级

4. **启动验证**
   - 每个服务启动时校验：
     - 能否从 JWKS 端点获取至少一个有效公钥
     - 公钥算法是否为 RS256
     - 密钥是否处于有效期内（未过期、未 revoked）
   - 校验失败则服务启动失败（fail-fast）

## 理由

- **SOC2 合规**：90 天周期满足 SOC2 CC6.1 控制点要求
- **Vault 原生支持**：HashiCorp Vault 的 PKI / Transit 引擎支持自动轮换 + 宽限期配置
- **零停机轮换**：JWKS 多密钥机制允许新旧密钥并行，无需强制用户重新登录
- **fail-fast 降低运行时风险**：启动时即发现密钥问题，避免运行时认证大面积失败

## 影响

- **对现有代码的影响**
  - `schemaplexai-gateway` 需新增 JWKS 端点及密钥管理逻辑
  - 所有服务的 JWT 验证逻辑需从静态公钥改为 JWKS 动态获取
  - 移除所有配置文件中的 `jwt.public-key` 硬编码项

- **对运维的影响**
  - Vault 需配置 90 天自动轮换策略 + 7 天宽限期
  - 监控告警：轮换失败、JWKS 端点不可用、密钥即将过期（提前 14 天预警）

- **对性能的影响**
  - JWKS 公钥可本地缓存（TTL = 1 小时），验证性能无显著下降
  - 首次启动增加一次 JWKS 请求（~5-10ms）

## 替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 180 天轮换 | 运维负担低 | 不满足 SOC2 要求 | 拒绝 |
| 30 天轮换 | 泄露窗口极小 | 运维成本高，Vault 负载大 | 拒绝 |
| **90 天轮换（本方案）** | 平衡安全与运维 | 需 Vault 集成 | **采纳** |
| 不轮换，依赖撤销列表 | 实现简单 | JWT 无状态优势丧失，列表膨胀 | 拒绝 |
