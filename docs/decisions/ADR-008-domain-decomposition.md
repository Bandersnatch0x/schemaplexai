# ADR-008: 按域拆分 Service 层

> **日期**: 2026-04-29
> **决策人**: 架构评审委员会
> **状态**: 已批准

---

## 背景

原 `schemaplexai-service` 包含 8 个业务域，随着功能增长将成为大泥球：
- 不同业务域的发布节奏被绑定
- Agent 引擎高负载影响系统治理接口
- 无法针对计算密集型服务（agent-engine）单独扩缩容

## 决策

按业务域拆分为 **9 个独立服务模块**（可独立部署），保留 `schemaplexai-web` 作为 BFF 层。

## 理由

1. **单一职责**：每个服务只负责一个业务域，代码内聚度高
2. **独立扩缩容**：agent-engine 可单独扩容，ops/quality 等非核心服务可缩减
3. **故障隔离**：集成服务（外部网络依赖多）故障不影响核心 Agent 引擎
4. **团队并行开发**：不同团队独立开发、测试、部署各自服务

## 影响

- **正面**：部署灵活、故障隔离、团队并行度提升
- **负面**：增加服务治理复杂度（需服务发现、配置中心）；跨服务调用增加网络延迟
- **缓解**：OpenFeign 连接池优化 + Caffeine 本地缓存；非必要不拆分查询链路

## 替代方案

| 方案 | 评估 | 结论 |
|------|------|------|
| 保持单模块 | 部署简单，但无法解决团队并行度和故障隔离 | 拒绝 |
| 拆分为 5 个粗粒度服务 | 减少运维复杂度，但 Agent 引擎与配置未分离 | 拒绝 |
| 拆分为 15+ 细粒度服务 | 过度拆分，增加网络延迟和运维负担 | 拒绝 |
| **9 个业务域服务（本方案）** | 按限界上下文拆分，平衡独立性与复杂度 | **采纳** |
