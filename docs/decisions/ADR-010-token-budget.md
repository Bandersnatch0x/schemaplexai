# ADR-010: Token 预算管理

> **日期**: 2026-04-29
> **决策人**: 架构评审委员会
> **状态**: 已批准

---

## 背景

LLM 调用成本高昂，缺乏执行级 Token 管控手段：
- 单执行可能消耗数万 Token，导致账单失控
- 无预警机制，超限后才能事后发现
- 不同租户/Agent 的预算需求不同，需要精细化控制

## 决策

引入 `TokenBudget` 机制，每轮 LLM 调用前检查预算，超限时触发压缩或终止。

## 理由

1. **防止单执行消耗过多 Token**：预算上限强制拦截异常高消耗执行
2. **支持按租户/Agent 精细化成本控制**：预算参数可配置化绑定到 Agent 配置
3. **提前预警避免账单失控**：输入 Token 预检 + 输出 Token 后扣，实时跟踪消耗

## 影响

- **正面**：成本可控、租户级配额可落地
- **负面**：Token 预估逻辑可能不准确，需预留缓冲；增加预算消耗存储到执行记录
- **缓解**：预留 20% 缓冲；实际消耗校准预估模型；超限时先压缩上下文再终止

## 替代方案

| 方案 | 评估 | 结论 |
|------|------|------|
| 仅事后统计，不事中拦截 | 实现简单，但无法防止单次执行超支 | 拒绝 |
| 按时间窗口限制（如每秒 Token 数） | 无法限制单执行总消耗 | 拒绝 |
| **每轮调用前预检 + 超限时压缩/终止（本方案）** | 事中拦截 + 渐进式降级，平衡成本与体验 | **采纳** |
