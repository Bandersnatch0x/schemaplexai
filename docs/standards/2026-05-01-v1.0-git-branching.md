# Git 分支策略与合并规范

> **主题**: `git-branching`
> **阶段**: `standard`
> **版本**: v1.0
> **状态**: 已批准
> **日期**: 2026-05-01

---

## 1. 分支模型

采用 **Git Flow 简化版**，保留 `main` 和 `develop` 两条长期分支：

```
main        ●────●────●────●────────────────────●────●
            ↑    ↑         ↑                    ↑    ↑
         v1.0  hotfix   v1.1(tag)            v1.2  hotfix

develop     ●────●────●────●────●────●────●────●────●
                 ↑         ↑    ↑              ↑
              feature/a  feature/b  feature/c   release
```

### 分支类型

| 分支 | 生命周期 | 来源 | 合并目标 | 命名规范 |
|------|---------|------|---------|---------|
| `main` | 长期 | — | — | — |
| `develop` | 长期 | — | — | — |
| Feature | 临时 | `develop` | `develop` | `feature/<topic>-<ticket-id>` |
| Fix | 临时 | `develop` | `develop` | `fix/BUG-NNN` |
| Hotfix | 临时 | `main` | `main` + cherry-pick to `develop` | `hotfix/BUG-NNN-main` |
| Release | 临时 | `develop` | `main` + merge back to `develop` | `release/vX.Y.Z` |

## 2. 主分支保护规则

### 强制约束

| 规则 | 说明 | 适用分支 |
|------|------|---------|
| **禁止直接 push** | 所有合并必须通过 Pull Request | `main`, `develop` |
| **禁止 force push** | `git push --force` 永久禁止 | `main`, `develop` |
| **Status checks 必过** | CI / 测试 / 覆盖率门禁失败则阻塞合并 | `main`, `develop` |
| **至少 1 人 Review** | Code Review 通过后方可合并 | `main`, `develop` |
| **Hotfix 需 2 人 Review** | P0/P1 修复需模块 Owner + 技术负责人 | `main` |

### main 分支额外约束

- 仅接受来源：`release/*`、`hotfix/*`
- 不接受任何 `feature/*` 或 `fix/*` 分支的直接合并
- 每次合并必须打语义化版本 tag（`vX.Y.Z`）

## 3. 并行开发同步策略

### 3.1 Feature 分支日常同步

当 `develop` 上有新合并时，所有并行开发的 Feature 分支必须及时同步：

```bash
# 推荐方式：rebase 保持 linear history
git fetch origin
git rebase origin/develop

# 如果 rebase 冲突频繁，允许使用 merge（需团队共识）
git fetch origin
git merge origin/develop
```

**频率要求**：
- 开发周期 ≤ 3 天：至少同步 1 次（提交 PR 前）
- 开发周期 > 3 天：每 2 天至少同步 1 次
- 提交 PR 前：必须同步到最新 `develop`

### 3.2 冲突预防机制

1. **Plan 阶段识别冲突点**：Graphify task graph 中标注跨分支文件依赖
2. **模块 Owner 协调**：同一模块有多个 Feature 并行时，模块 Owner 负责合并顺序
3. **先行合并原则**：被依赖的 Feature 优先合并，依赖方随后 rebase

### 3.3 冲突解决流程

```
Feature A ──→ develop（先合并）
                    ↑
Feature B ──────────┘（后 rebase，解决冲突）
```

1. 冲突发生时，由**后合并**的分支负责人解决
2. 若冲突涉及架构/接口变更，冲突双方 + 模块 Owner 三方确认方案
3. 禁止在冲突解决中引入未经评审的代码变更（只解决冲突，不做重构）

## 4. PR/MR 合并规范

### 4.1 PR 创建要求

- **标题**: 遵循 Conventional Commits（`feat:`, `fix:`, `refactor:` 等）
- **描述**: 必须包含变更摘要、测试计划、关联 Ticket
- **基线分支**: 确认目标分支正确（`feature/*` → `develop`，`hotfix/*` → `main`）

### 4.2 合并方式

| 方式 | 使用场景 | 说明 |
|------|---------|------|
| **Squash and Merge** | Feature / Fix 分支 | 将多个 commit 压缩为一个，保持 `develop` 历史整洁 |
| **Rebase and Merge** | 个人独立分支（无协作者） | 保留完整 commit 历史，但要求已 rebase 到最新基线 |
| **Create Merge Commit** | Release / Hotfix 分支 | 保留分支合并痕迹，便于追溯 |

**默认推荐**：`Squash and Merge`（Feature/Fix），合并消息使用 PR 标题。

### 4.3 合并前检查单

- [ ] 分支已同步到最新目标分支（`develop` 或 `main`）
- [ ] CI 全部通过（构建、测试、覆盖率 ≥ 80%）
- [ ] Code Review 通过（至少 1 人，Hotfix 需 2 人）
- [ ] 无未解决的冲突标记
- [ ] 关联的 Spec/Plan 文档状态已更新

## 5. Release 流程

```
develop ──→ release/v1.1.0 ──→ main（tag v1.1.0）──→ merge back ──→ develop
                ↑                                    ↓
         仅 bugfix，               生产部署后打 tag，
         不新增 feature            然后合并回 develop
```

1. **创建 Release 分支**：从最新 `develop` 切出 `release/vX.Y.Z`
2. **冻结特性**：Release 分支仅接受 bugfix，不接受新功能
3. **合并到 main**：通过 PR 合并，合并后打 tag `vX.Y.Z`
4. **回写 develop**：将 Release 分支合并回 `develop`，保持同步
5. **删除 Release 分支**：回写完成后删除

| 现有流程 | 分支操作 | 对应规范 |
|---------|---------|---------|
| `feature-workflow.md` Phase 7 | `git checkout -b feature/<topic>` | 分支命名规范 |
| `feature-workflow.md` Phase 10 | 合并到 `develop` | 禁止直接 push，必须 PR |
| `fix-workflow.md` Phase 6 | `fix/BUG-NNN-develop` | 常规修复走 `develop` |
| `fix-workflow.md` Phase 8 | `hotfix/BUG-NNN-main` | Hotfix 双写 + 2 人 Review |
| `CLAUDE-DEVELOPER.md` Apply | `EnterWorktree` 隔离 | 复杂变更（≥3 模块）建议 worktree |

---

## 7. 禁止行为

- **禁止** Feature 分支直接合并到 `main`
- **禁止** 在 `main` 或 `develop` 上直接 commit
- **禁止** 使用 `git push --force` 到任何长期分支
- **禁止** 在冲突解决中夹带未经评审的代码变更
- **禁止** PR 合并后保留远程 Feature 分支（合并后自动或手动删除）

---

## 8. 异常处理

| 场景 | 处理方式 |
|------|---------|
| Feature 分支已过时，rebase 冲突极多 | 放弃分支，从最新 `develop` 重建，手动 cherry-pick 有效 commit |
| Hotfix cherry-pick 到 `develop` 冲突 | 在 `develop` 上创建对应 `fix/BUG-NNN-develop` 分支手动解决 |
| 误操作 force push 到 `main` | 立即通知团队，由仓库管理员恢复，审查是否有代码丢失 |
| 紧急绕过 CI 合并（生产事故） | 事后 24h 内补充说明文档，经技术负责人书面批准 |
