## 1. Spec 评审检查单

### 1.1 完整性

- [ ] 是否覆盖了 PRD/需求中的所有验收标准？
- [ ] 是否有遗漏的边界条件？
- [ ] 是否定义了错误处理策略？
- [ ] 是否包含性能指标？

### 1.2 一致性

- [ ] 与现有系统架构是否一致？
- [ ] 命名规范是否符合项目标准？
- [ ] 数据模型是否与现有表结构兼容？
- [ ] API 风格是否与其他模块统一？

### 1.3 可执行性

- [ ] 接口定义是否清晰？（参数、返回值、异常）
- [ ] 状态转换是否完整？（所有状态 × 所有事件）
- [ ] 算法/策略是否有伪代码或流程图？
- [ ] 一个合格的开发能否直接按 Spec 实现？

### 1.4 安全性

- [ ] 是否涉及用户输入？输入验证策略是什么？
- [ ] 是否涉及权限控制？鉴权逻辑在哪里？
- [ ] 是否涉及敏感数据？加密/脱敏策略是什么？
- [ ] 是否有 OWASP Top 10 风险？

### 1.5 性能与可扩展性

- [ ] 数据库查询是否可索引？
- [ ] 是否有缓存策略？
- [ ] 并发场景是否考虑？（锁、CAS、事务隔离级别）
- [ ] 大数据量场景是否考虑？（分页、分批、流式）

### 1.6 兼容性

- [ ] 是否破坏现有 API？（版本控制策略）
- [ ] 是否需要数据库迁移脚本？
- [ ] 是否需要前端配合修改？
- [ ] 是否有降级/回退方案？

### 1.7 评审结论

| 结论 | 说明 | 下一步 |
|------|------|--------|
| **通过** | 无需修改 | 进入 Plan 阶段 |
| **有条件通过** | 小修改即可 | 修改后无需再次评审 |
| **不通过** | 重大缺陷 | 修改后重新评审 |

## 2. Plan 评审检查单

### 2.1 任务完整性

- [ ] 是否覆盖了 Spec 中的所有要求？
- [ ] 是否有遗漏的依赖项？（配置、权限、环境）
- [ ] 是否包含测试任务？
- [ ] 是否包含文档更新任务？

### 2.2 任务粒度

- [ ] 单个任务是否能在 4 小时内完成？
- [ ] 任务描述是否清晰到可以独立执行？
- [ ] 每个任务是否有明确的验收标准？

### 2.3 依赖与顺序

- [ ] 任务顺序是否正确？
- [ ] 是否存在循环依赖？
- [ ] 关键路径上的任务是否有缓冲时间？

### 2.4 工期合理性

- [ ] 估算是否过于乐观？（建议 × 1.5 缓冲）
- [ ] 是否考虑了会议、评审、等待时间？
- [ ] 是否与里程碑冲突？

### 2.5 风险与回退

- [ ] 是否识别了主要风险？
- [ ] 风险缓解措施是否可操作？
- [ ] 回退方案是否切实可行？

### 2.6 评审结论

| 结论 | 说明 | 下一步 |
|------|------|--------|
| **通过** | 无需修改 | 进入开发阶段 |
| **有条件通过** | 小修改即可 | 修改后无需再次评审 |
| **不通过** | 任务分解不合理 | 重新分解后再次评审 |

## 3. Code Review 检查单

### 3.1 代码正确性

- [ ] 代码是否确实解决了问题？（对照 Bug 描述或 Spec）
- [ ] 是否有明显的逻辑错误？
- [ ] 边界条件是否处理正确？
- [ ] 错误处理是否完善？（无吞异常、无裸 catch）

### 3.2 代码质量

- [ ] 函数是否聚焦？（< 50 行）
- [ ] 文件是否内聚？（< 800 行）
- [ ] 命名是否清晰？（无缩写、无歧义）
- [ ] 是否有重复代码？（DRY 原则）
- [ ] 是否有魔法数字？（应使用常量）
- [ ] 嵌套是否过深？（> 4 层需重构）

### 3.3 安全性

- [ ] **无硬编码密钥/密码/Token**
- [ ] 用户输入是否验证？
- [ ] SQL 查询是否参数化？
- [ ] 文件路径是否转义？
- [ ] 是否有 XSS/CSRF 风险？
- [ ] 敏感数据是否脱敏/加密？

### 3.4 性能

- [ ] 数据库查询是否有 N+1 问题？
- [ ] 是否有未分页的查询？
- [ ] 缓存使用是否合理？
- [ ] 是否有内存泄漏风险？
- [ ] 大数据量操作是否分批？

### 3.5 测试

- [ ] 是否有新功能的单元测试？
- [ ] 测试是否覆盖了边界条件？
- [ ] 测试是否独立？（无测试间依赖）
- [ ] 测试命名是否描述行为？
- [ ] 测试覆盖率是否 ≥ 80%？

### 3.6 文档与注释

- [ ] 公共 API 是否有 JavaDoc？
- [ ] 复杂算法是否有注释说明？
- [ ] 是否有无意义的注释？（注释应解释 Why，不是 What）
- [ ] 变更日志是否更新？

### 3.7 提交规范

- [ ] 提交信息是否符合 Conventional Commits？
- [ ] 单次提交是否原子？（一个提交只做一件事）
- [ ] 提交是否包含无关变更？

### 3.8 评审严重等级

| 等级 | 含义 | 操作 |
|------|------|------|
| **CRITICAL** | 安全漏洞 / 数据丢失风险 / 明显 Bug | **必须修复** |
| **HIGH** | 逻辑缺陷 / 严重性能问题 / 测试缺失 | **必须修复** |
| **MEDIUM** | 可维护性问题 / 设计建议 | **建议修复** |
| **LOW** | 代码风格 / 命名建议 / 优化建议 | **可选修复** |

### 3.9 评审结论

- [ ] 所有 CRITICAL 和 HIGH 问题已修复
- [ ] CI 全部通过
- [ ] 测试覆盖率达标

## 4. 安全评审触发条件

以下变更 **必须** 经过安全评审（使用 `security-reviewer` agent）:

- [ ] 认证/授权代码变更
- [ ] 用户输入处理逻辑变更
- [ ] 数据库查询变更（尤其是动态 SQL）
- [ ] 文件系统操作变更
- [ ] 外部 API 调用变更
- [ ] 加密/解密逻辑变更
- [ ] 支付/财务相关代码
- [ ] 配置文件变更（尤其涉及密钥）

## 5. 快速参考卡

### Feature 评审流程
```
Spec 评审 → Plan 评审 → Code Review
   ↑           ↑           ↑
  2工作日     1工作日     1工作日（时间盒）
```

### Fix 评审流程
```
Plan 评审 → Code Review
   ↑           ↑
 P0/P1:1h   P0/P1:2人Review
 P2/P3:1d
```

### 评审底线
- **绝不合并**: 包含 CRITICAL 问题的代码
- **不合并**: 包含未修复 HIGH 问题的代码（必须修复后才能合并）
- **可以合并**: 仅包含 MEDIUM/LOW 问题的代码（需创建后续优化任务）
