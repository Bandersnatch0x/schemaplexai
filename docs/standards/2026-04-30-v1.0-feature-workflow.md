# Feature 开发工作流程

> **主题**: `feature-workflow`
> **阶段**: `standard`
> **版本**: v1.0
> **状态**: 已批准
> **日期**: 2026-04-30

---

## 1. 流程概览

```
需求澄清 → 编写 Spec → 评审 Spec →  Overview 当前实现
                                              ↓
                                    编写 Plan → 评审 Plan → 开发实现
                                                                    ↓
                                                           自测 → Code Review → 合并
```

**任何新功能或超过 50 行代码的变更必须遵循此流程。**

### 与 .claude/ 执行层的关系

本规范定义了**权威流程**（门禁、评审标准、产出要求）。实际执行时，Claude Code 在 `.claude/changes/<feat>/` 工作区中按六阶段驱动：

```
Propose → Spec → Design → Plan → Apply → Archive
  ↑        ↑       ↑       ↑      ↑        ↑
.claude/ 执行层加速    →    docs/ 权威沉淀
```

详见 `.claude/CLAUDE-DEVELOPER.md`。两者的映射关系：

| 本规范阶段 | .claude/ 阶段 | 产出位置 |
|-----------|--------------|---------|
| Phase 1-2: 需求澄清 + 编写 Spec | **Propose** + **Spec** | `.claude/changes/<f>/proposal.md` + `spec.md` |
| Phase 3: 评审 Spec | **Spec 门禁** | 评审通过后可选沉淀到 `docs/specs/` |
| Phase 4-5: Overview + 设计 | **Design** | `.claude/changes/<f>/design.md` → `docs/designs/` |
| Phase 6-7: 编写 Plan + 评审 | **Plan** | `.claude/changes/<f>/tasks.md`（Graphify 图结构） |
| Phase 8-9: 开发 + 自测 | **Apply** | 代码 + 测试 |
| Phase 10: 合并归档 | **Archive** | `.claude/changes/archive/` + `docs/` 更新 |

**原则**: `.claude/` 是执行加速器，`docs/` 是权威基线。执行层不替代本规范，而是提供结构化加速。

## 2. 阶段详解

### Phase 1: 需求澄清

**触发条件**: 产品需求（PRD）已创建或用户故事已确认

**执行人**: 产品经理 + 技术负责人 + 开发工程师

**产出**:
- 明确的验收标准（Acceptance Criteria）
- 功能边界（做什么 / 不做什么）
- 依赖关系清单

**检查点**:
- [ ] 需求是否涉及多模块变更？
- [ ] 是否有技术预研需求（新框架/新中间件）？
- [ ] 是否需要新增数据库表/索引？
- [ ] 是否涉及安全敏感操作（auth、支付、文件上传）？

---

### Phase 2: 编写 Spec（技术规格说明书）

**触发条件**: 需求澄清完成，技术方案已初步构思

**执行人**: 开发工程师（主笔）+ 技术负责人（审核）

**文档位置**: `docs/specs/<topic>.md`

**必须包含的内容**:

| 章节 | 说明 | 必填 |
|------|------|------|
| 1. 概述 | 本 Spec 解决的问题、范围、非目标 | 是 |
| 2. 架构视图 | 组件关系图、数据流图 | 是 |
| 3. 核心组件规格 | 每个类的职责、接口定义、状态机 | 是 |
| 4. API 接口 | 请求/响应格式、错误码、示例 | 是 |
| 5. 数据模型 | 表结构、字段类型、索引、约束 | 是 |
| 6. 非功能需求 | 性能指标、安全要求、兼容性 | 是 |
| 7. 风险与回退 | 已知风险、降级方案 | 是 |
| 8. 相关文档 | 关联的 ADR、Design、Plan | 否 |

**模板**: 使用 `docs/DOCUMENT-TEMPLATE.md`

**规模评估**:
- 预计代码变更 < 50 行：可省略 Spec，但需在设计文档中记录
- 预计代码变更 50-200 行：精简 Spec（概述 + API + 数据模型）
- 预计代码变更 > 200 行：完整 Spec

---

### Phase 3: 评审 Spec

**触发条件**: Spec 初稿完成

**评审人**: 技术负责人（必须）+ 架构师（涉及架构变更时）+ 安全负责人（涉及安全时）

**评审方式**: 异步文档评审（PR Review 模式）或会议评审（复杂功能）

**评审检查单**（详见 `docs/standards/review-checklists.md`）:

- [ ] **完整性**: 是否覆盖了所有验收标准？
- [ ] **一致性**: 与现有架构、命名规范、数据模型是否一致？
- [ ] **可执行性**: 接口定义是否清晰到可以直接实现？
- [ ] **安全性**: 是否考虑了输入验证、权限控制、数据隔离？
- [ ] **性能**: 是否评估了数据库查询、缓存策略、并发影响？
- [ ] **兼容性**: 是否影响现有 API/数据？如何迁移？

**评审结果**:
- **通过**: 进入 Phase 4
- **有条件通过**: 记录修改项，修改后无需再次评审
- **不通过**: 重大修改后重新评审

**时间盒**: 2 个工作日内必须完成评审反馈

---

### Phase 4: Overview 当前实现

**触发条件**: Spec 评审通过

**执行人**: 开发工程师

**目标**: 了解现有代码状态，识别复用点、冲突点、技术债务

**操作清单**:

1. **代码扫描**
   ```bash
   # 查找相关现有代码
   grep -rn "关键词" --include="*.java" schemaplexai-目标模块/src/
   grep -rn "关键词" --include="*.tsx" schemaplexai-ui/src/
   ```

2. **依赖分析**
   - 本模块已有哪些类可以直接复用？
   - 是否需要修改现有接口？是否破坏向后兼容？
   - 跨服务调用是否新增？是否需要更新 Feign 客户端？

3. **技术债务识别**
   - 现有代码中是否有 TODO 与本功能相关？
   - 是否需要先重构再实现？
   - 是否有硬编码值需要提取为配置？

4. **测试现状**
   - 现有模块是否有测试？测试覆盖率如何？
   - 本功能需要哪些类型的测试（单元/集成/E2E）？

**产出**: `Overview 报告`（可写入 Plan 文档的"现状分析"章节）

---

### Phase 5: 编写 Plan（实施计划）

**触发条件**: Overview 完成

**执行人**: 开发工程师（主笔）+ 技术负责人（审核）

**文档位置**: `docs/plans/<topic>-plan.md`

**必须包含的内容**:

| 章节 | 说明 |
|------|------|
| 1. 目标 | 本 Plan 要达成的具体目标 |
| 2. 现状分析 | Overview 结果：已有代码、复用点、技术债务 |
| 3. 任务分解 | 按文件/按模块拆分，每个任务含：文件、变更类型、验收标准 |
| 4. 依赖关系 | 任务间的先后顺序、外部依赖 |
| 5. 工期估算 | 每个任务的预估工时 |
| 6. 风险与缓解 | 实现风险、测试风险、部署风险 |
| 7. 回退方案 | 如果延期或失败，如何回退？ |

**任务粒度要求**:
- 每个任务应能在 **4 小时内**完成
- 每个任务必须有明确的 **验收标准**（什么算完成）
- 使用 checkbox 语法 `- [ ]` 跟踪进度

**Plan 评审前置条件**:
- [ ] Spec 已通过评审
- [ ] Overview 报告已记录
- [ ] 任务分解粒度合理

---

### Phase 6: 评审 Plan

**触发条件**: Plan 初稿完成

**评审人**: 技术负责人（必须）+ 相关模块 Owner

**评审检查单**:

- [ ] **任务完整性**: 是否覆盖了 Spec 中的所有要求？
- [ ] **粒度合理性**: 单个任务是否能在 4 小时内完成？
- [ ] **依赖正确性**: 任务顺序是否正确？是否存在循环依赖？
- [ ] **工期合理性**: 估算是否过于乐观？是否包含缓冲？
- [ ] **测试覆盖**: 每个任务是否对应测试任务？
- [ ] **回退可行性**: 回退方案是否可操作？

**评审结果**:
- **通过**: 进入 Phase 7
- **有条件通过**: 记录修改项
- **不通过**: 重新分解后再次评审

---

### Phase 7: 开发实现（Do）

**触发条件**: Plan 评审通过

**执行规范**:

1. **分支管理**
   ```bash
   git checkout -b feature/<topic>-<ticket-id>
   ```

2. **开发顺序**
   - 先写测试（TDD: RED → GREEN → REFACTOR）
   - 再写实现
   - 最后集成测试

3. **代码规范**
   - 遵循 `docs/standards/coding-style.md`
   - 函数 < 50 行
   - 文件 < 800 行
   - 无深度嵌套（> 4 层）
   - 不可变性优先

4. **提交规范**
   ```
   feat: 新增 Agent 执行中断功能

   - 实现 PAUSED 状态处理器
   - 添加快照持久化
   - 补充单元测试

      ```

5. **质量门禁**（代码变更 > 30 行时触发）
   - `/verify-change` → 分析变更影响
   - `/verify-quality` → 检查复杂度、重复代码
   - `/verify-security` → 扫描安全漏洞

**禁止行为**:
- 未经 Plan 评审直接开发
- 在 feature 分支中修改与本功能无关的代码
- 提交包含硬编码密钥或密码的代码
- 跳过测试直接提交

---

### Phase 8: 自测

**执行人**: 开发工程师

**检查单**:
- [ ] 单元测试全部通过
- [ ] 集成测试通过（涉及跨服务时）
- [ ] 手动走通主流程
- [ ] 边界条件验证（空输入、超大输入、并发）
- [ ] 多租户隔离验证

---

### Phase 9: Code Review

**执行人**: 模块 Owner 或指定 Reviewer

**评审检查单**: 详见 `docs/standards/review-checklists.md`

**必须通过项**:
- [ ] 无 Critical/High 安全问题
- [ ] 测试覆盖率 ≥ 80%
- [ ] 所有 CI 检查通过
- [ ] 文档同步更新（Spec/Plan/API 文档）

---

### Phase 10: 合并与归档

1. 合并到 develop 分支
2. 更新 Spec 状态为 `已实现`
3. 更新 Plan 状态为 `已完成`
4. 归档 Plan 到 `docs/archive/`
5. 更新 `docs/README.md` 索引

---

## 4. 时间盒

| 阶段 | 时间盒 | 说明 |
|------|--------|------|
| 需求澄清 | 1-2d | 复杂功能可延长 |
| 编写 Spec | 0.5-2d | 视功能复杂度 |
| 评审 Spec | 2 工作日 | 必须在此时间内反馈 |
| Overview | 0.5-1d | 熟悉现有代码 |
| 编写 Plan | 0.5-1d | 视任务拆分复杂度 |
| 评审 Plan | 1 工作日 | 轻量评审 |
| 开发实现 | 按 Plan | — |
| Code Review | 1 工作日 | 必须在此时间内反馈 |
